<h2>О компании</h2>

«Тёплый дом» — это небольшая компания, которая организует удалённое управление отоплением в доме. Недавно она выиграла тендер и получила заказ на создание экосистемы умных посёлков на территории нескольких регионов страны.
Для выполнения заказа компании нужно расширить функционал своей системы и масштабировать её. Пользователи должны иметь возможность управлять не только отоплением, но и умными устройствами в доме. Чтобы подключиться к экосистеме, в домах должны быть установлены специальные датчики и реле. Сейчас они есть только в половине домов, которые хотят подключиться.
Состояние компании в настоящий момент не позволяет в полной мере реализовать новые бизнес-цели. Для этого требуется пересмотр и оптимизация всей экосистемы.

ФТ TO-BE
1) Пользователи через веб-сайт или мобильное приложение могут управлять устройствами умного дома, такими как отопление, освещение, автоматические ворота, может настраивать сценарии работы и просматривает телеметрию
2) Пользователь самостоятельно выбирает необходимые ему модули умного дома (устройства), сам их подключает
3) Пользователь может просматривать записи с камер видеонаблюдения подключенных к умному дому

НФТ TO-BE
1) В описании сказано что компании необходимо масштабироваться на поселки нескольких регионов страны. Возьмем в расчет что в каждом регионе нужно подключить по одному поселку. Поселки по численности в среднем от 1000чел (средние поселки) до 3000чел (большие поселки). В каждом доме живет от 2 до 4 чел. Сказано что поселки в нескольких регионах -  предположим что больше 1, но не более 5 регионов. При округленном среднем подсчете предполагаем, что нужно подключить около 2000 домов.
2) Экосистема должна быть доступна пользователю в режиме самообслуживания по модели SaaS.
3) Датчики подключаются к модулю управления умным домом, необходим интернет канал до модуля управления.
4) Модуль управления умным домом должен дублировать все настройки сценариев управления датчиков и реле на случай обрыва интернет канал до сервера компании.
5) Latency включение/выключения устройств при управлении через мобильное приложение не более 1-2 сек., т.к. предусматривается управление устройствами которым необходим быстрый отклик (освещение, запирать и отпирать автоматические ворота)
6) Обеспечение безопасного подключения к системе управления умным домом

# Задание 1. Анализ и планирование

### 1. Описание функциональности монолитного приложения

Текущее приложение компании позволяет только управлять отоплением в доме и проверять температуру.

**Управление отоплением:**

- Пользователи могут через приложение управлять отопление дома, включать и выключать его, настраивать температуру включения/выключения (хотя об этом не сказано, но т.к. реле отопления в одной экосистеме с датчиками то думаю это реализовано).
- Система поддерживает только определенные реле управления
- Подключение новых отопительных приборов возможен только через вызов специалиста компании.

**Мониторинг температуры:**

- Пользователи могут через приложение узнать температуру в доме.
- Система поддерживает определенные датчики температуры
- Подключение новых датчиков температуры возможен только через вызов специалиста компании

### 2. Анализ архитектуры монолитного приложения

Архитектура приложения представляет из себя монолит на Go с СУБД Postgres. Всё синхронно. Никаких асинхронных вызовов, микросервисов и реактивного взаимодействия в системе нет. Всё управление идёт от сервера к датчику. Данные о температуре также получаются через запрос от сервера к датчику. Приложение пользователей так же взаимодействует с данным монолитом.

Сейчас развёрнуто монолитное приложение с 100 веб-клиентами, к системе подключены 100 модулей управления отоплением

### **4. Проблемы монолитного решения**

- Нет отказоустойчивости управления умным домом, если основной сервер приложения выходит из строя и RTO большое, то при включенных ранее реле отопления дома их никто не сможет выключить что может привести к пожару.
- Все запросы обрабатываются синхронно, что ведет к гонке ресурсов при большом кол. подключенных умных устройств и авторизированных пользователей.
- Расширение функциональности приложения приведет к частым деплоям фич и частой остановке монолита, что скажется на времени
- Монолит можно масштабировать только вертикально, при дальнейшего расширение клиентов можно упереться в ресурсы. Так же большие монолиты сложно и дорого бэкапить, резервировать.
- Невозможно масштабировать отдельные функциональные модули приложения, как например функциональность управление устройствами.
- В случае доработки функциональности веб-клиента, или функциональности управления устройствами необходима перезагрузка всего монолита

Если вы считаете, что текущее решение не вызывает проблем, аргументируйте свою позицию.

Вам нужно:

1) сделать простое приложение temperature-api на любом удобном для вас языке программирования, которое при запросе /temperature?location= будет отдавать рандомное значение температуры.

Locations - название комнаты, sensorId - идентификатор названия комнаты

```
	// If no location is provided, use a default based on sensor ID
	if location == "" {
		switch sensorID {
		case "1":
			location = "Living Room"
		case "2":
			location = "Bedroom"
		case "3":
			location = "Kitchen"
		default:
			location = "Unknown"
		}
	}

	// If no sensor ID is provided, generate one based on location
	if sensorID == "" {
		switch location {
		case "Living Room":
			sensorID = "1"
		case "Bedroom":
			sensorID = "2"
		case "Kitchen":
			sensorID = "3"
		default:
			sensorID = "0"
		}
	}
```
